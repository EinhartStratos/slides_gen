# PRD：基于 FastAPI 的 SVG-PPT 生成服务

## 1. 文档目标

本文档用于定义新的 PPT 生成服务产品需求。

目标是将当前参考项目中的 PPT 生成能力，重构为一个**标准 FastAPI 异步服务**，并将模板处理与页面生成主链路从“规则 JSON + Mermaid 图片”升级为“SVG 模板 + SVG 可编辑图形 + SVG 转 PPTX”。

本文档只定义产品目标、用户场景、功能边界、核心流程、非功能需求和验收标准，不涉及具体代码实现细节。

## 2. 背景

现有参考实现主要有两条：

### 2.1 旧版生成链路

来源：`refs/pptx_gen`

特点：
- 通过模板解析器先把 PPT 模板解析成 JSON 规则
- 按页调用大模型生成文本、表格、Mermaid 图
- 将 Mermaid 渲染成图片
- 最后再组装成 PPT

当前问题：
- 模板主表示形式是 JSON，不利于后续直接复用 PPT 的视觉结构
- Mermaid 生成的是图片，不利于在 PowerPoint 中继续编辑
- 整体更偏脚本流程，不适合直接作为服务长期维护

### 2.2 SVG / PPTX 互转链路

来源：`refs/ppt-master`

特点：
- 能将 PPTX 直接转成逐页 SVG
- 能将 SVG 尽量转换回 PowerPoint 原生可编辑对象
- 更适合作为“模板可视结构”和“最终回写 PPT”的基础能力

## 3. 产品目标

新的服务需要满足以下目标：

- 提供标准 FastAPI 接口，支持异步调用生成 PPT
- 模板参数可选，不传时使用系统内置默认模板
- 模板主链路基于逐页 SVG，而不是先产出模板 JSON
- 保留“每页单独判断是否需要生成”的业务逻辑
- 图形页不再依赖 Mermaid，而是直接生成可编辑 SVG 图形
- 静态背景、图片、图标等尽量保留模板原样
- 最终通过 `svg_to_pptx` 生成可编辑 PPTX
- 项目代码结构完全采用标准 FastAPI 服务架构，不延续旧脚本式组织方式

## 4. 非目标

以下内容不在本阶段目标内：

- 不做前端页面或可视化后台
- 不做多租户权限系统
- 不做复杂工作流平台
- 不做任意 SVG 全量语法支持
- 不做对所有第三方 PPT 特性 100% 保真 round-trip
- 不做一次请求内同步等待所有页面生成完成的接口设计

## 5. 目标用户

### 5.1 直接用户

- 需要通过接口自动生成 PPT 的后端系统
- 需要根据业务需求批量生成汇报材料的服务调用方
- 需要将已有 PPT 模板服务化的研发团队

### 5.2 间接受益用户

- 最终需要拿到可编辑 PPT 的业务人员
- 需要后续修改图形、文案、结构的演示文稿使用者

## 6. 核心用户价值

新的服务要给用户带来 4 个核心价值：

- **更容易接入**：通过 HTTP API 触发生成，不再依赖本地脚本
- **更容易改**：图形尽量是 PPT 原生可编辑对象，不是整张图片
- **更稳定复用模板**：直接以 SVG 继承模板的页面结构与静态视觉元素
- **更适合扩展**：服务架构标准化，后续可加入队列、存储、监控和鉴权

## 7. 功能需求

## 7.1 模板选择

服务需要支持两类模板：

### 7.1.1 内置模板

- 系统内置至少 1 个默认模板
- 当请求不传模板参数时，自动使用默认模板
- 默认模板应等价于当前项目正在使用的模板

### 7.1.2 自定义模板

- 支持调用方上传自定义 PPTX 模板
- 模板导入后要转成系统可复用的模板工作区
- 后续生成任务可直接引用该模板

### 7.1.3 模板主表示形式

模板的**主表示形式必须是 SVG 工作区**，不是模板 JSON。

允许存在辅助元数据，但不能让“JSON 规则文件”成为生成主链路的唯一依据。

## 7.2 模板预处理

模板导入后，系统需要：

- 将 PPTX 转换为逐页 SVG
- 保留每页的视觉结构
- 区分静态背景内容与可编辑内容区域
- 为后续单页生成准备上下文

推荐行为：
- 使用 `pptx_to_svg` 生成 layered + flat 两类 SVG
- layered 结果用于机器理解结构
- flat 结果用于页面视觉预览和调试

## 7.3 单页独立判断逻辑

新服务必须保留旧逻辑中的关键原则：

- 每一页单独判断是否适合当前需求
- 不适合的页面必须输出“跳过”结果
- 跳过时要保留明确原因

即每页必须至少有以下生成判定信息：

- `page_no`
- `should_generate`
- `skip_reason`

## 7.4 页面生成方式

### 7.4.1 文本页

- 模型根据页面模板结构生成对应文本
- 文本要控制长度，避免溢出
- 支持标题、正文、分点、表格等常见结构

### 7.4.2 图形页

- 不再使用 Mermaid 生成图片
- 模型需要直接输出**可编辑 SVG 图形结构**
- 重点支持：
  - 架构图
  - 时序图
  - 流程图
  - 关系图
  - 分层模块图

### 7.4.3 静态内容保留

以下内容原则上应保留模板原样：

- 页面背景
- 装饰元素
- Logo
- 已有静态图标
- 静态图片
- 页脚、页码、统一风格元素

模型主要生成的应是：

- 可编辑文字
- 可编辑图形
- 可替换的业务标签
- 必要的连接线、节点、图例等结构化内容

## 7.5 SVG 元数据策略

关于“对应信息是否放在注释里”，产品要求如下：

### 7.5.1 不建议使用注释作为关键信息载体

不建议将关键结构信息放在 SVG 注释 `<!-- -->` 中，原因：

- 注释在解析链路中不稳定
- `svg_to_pptx` 当前解析方式不能保证后续稳定读取注释
- 注释不适合作为机器主数据结构

### 7.5.2 推荐方式

关键元数据应放在：

- 元素 `id`
- 元素 `data-*` 属性
- 页面级 `<metadata>` 节点

例如：
- `data-gen-role`
- `data-gen-binding`
- `data-gen-editable`
- `data-gen-source`
- `data-gen-page`

## 7.6 SVG 到 PPTX

系统需要将最终保留的页面 SVG 组织为标准项目工作区，然后调用 `svg_to_pptx` 生成 PPTX。

要求：
- 最终 PPT 尽量保持可编辑
- 支持将 SVG 形状转为原生 DrawingML 对象
- 对于不能安全转换的 SVG 内容，要在生成前校验并拒绝，而不是静默丢失

## 7.7 异步任务接口

考虑到中间存在多次大模型调用，接口层必须采用**异步任务模式**。

要求：
- 创建任务接口快速返回
- 后台继续执行模板处理、分页生成、SVG 校验和 PPTX 构建
- 支持任务状态轮询
- 支持任务结果下载

不建议的方式：
- 单个 HTTP 请求阻塞 1~10 分钟等待最终文件

## 8. 业务流程

## 8.1 主流程

```text
提交需求 -> 选择默认模板或自定义模板 -> 模板预处理/复用模板工作区
-> 按页分析 -> 每页判断是否生成 -> 生成页面 SVG
-> 校验 SVG 是否符合 PPT 可转换约束 -> 汇总保留页面
-> 调用 svg_to_pptx -> 输出 PPTX -> 返回任务结果
```

## 8.2 单页处理流程

```text
读取模板页 SVG
-> 提取本页结构与静态元素
-> 结合需求判断 should_generate
-> 如果 false，记录 skip_reason 并跳过
-> 如果 true，生成目标 SVG 页面
-> 校验 SVG 语法与受控子集
-> 写入最终工作区
```

## 9. 功能边界与约束

## 9.1 受控 SVG 子集

为了保证 `svg_to_pptx` 能稳定转换，模型输出必须限制在受控 SVG 子集内。

优先支持的元素：
- `svg`
- `g`
- `rect`
- `circle`
- `ellipse`
- `line`
- `polyline`
- `polygon`
- `path`
- `text`
- `tspan`
- `image`

需要谨慎或尽量避免的能力：
- 复杂滤镜
- marker 箭头强依赖
- 任意脚本
- foreignObject
- 不受控的 CSS
- 复杂 mask / clipPath 叠套

## 9.2 图形生成边界

本阶段图形生成聚焦于：

- 架构图
- 时序图
- 流程图
- 模块关系图

本阶段不强求支持：
- 复杂统计图表自动计算引擎
- 任意矢量插画
- 复杂 3D、发光、滤镜、渐变特效

## 9.3 模板兼容性边界

不是所有 PPT 模板都能无损转换为后续可编辑 SVG 工作区。

本阶段优先支持：
- 常规文本框
- 常规表格
- 常规形状
- 普通图片
- 结构清晰的模板页

需要后续专项验证的模板特性：
- SmartArt
- OLE 对象
- 复杂嵌套组合形状
- 特殊动画
- 特殊图表对象

## 10. 非功能需求

## 10.1 性能

- 单次任务支持多页并发生成
- 支持限制最大并发页数，防止大模型调用失控
- 模板预处理结果应支持缓存复用

## 10.2 稳定性

- 任意单页失败不能直接导致整个服务进程崩溃
- 必须能返回清晰的任务失败原因
- 中间产物可选保留，便于排查

## 10.3 可维护性

- 采用标准 FastAPI 分层架构
- 业务流程、适配器、模型定义分层清楚
- 便于未来替换 LLM 提供方或任务队列

## 10.4 可观测性

- 记录任务级日志
- 记录页面级日志
- 记录模板处理阶段耗时
- 记录模型调用耗时与失败原因
- 记录 SVG 校验失败详情

## 10.5 安全性

- API Key 不能写死在代码中
- 自定义模板上传要做文件类型检查
- 工作目录隔离，防止任务之间相互污染

## 11. 核心产物

每次任务建议输出以下产物：

- 原始请求快照
- 选定模板信息
- 每页判定结果
- 生成后的每页 SVG
- SVG 校验报告
- 最终 PPTX
- 任务汇总清单

## 12. 关键指标

第一阶段建议关注以下指标：

- 任务成功率
- 单页生成成功率
- SVG 校验通过率
- 最终 PPTX 可打开率
- 图形页可编辑率
- 平均任务耗时
- 模板复用命中率

## 13. 验收标准

满足以下条件可认为 PRD 对应目标达成：

### 13.1 服务形态

- 提供 FastAPI 接口
- 接口采用异步任务模式
- 支持任务提交、状态查询、结果下载

### 13.2 模板能力

- 不传模板参数时自动使用内置默认模板
- 支持导入自定义 PPTX 模板
- 模板主链路基于 SVG 页面，而不是先转规则 JSON

### 13.3 生成能力

- 保留逐页 `should_generate` 判定逻辑
- 文本页可正常生成
- 图形页不再依赖 Mermaid 图片
- 架构图 / 时序图 / 流程图可生成受控 SVG

### 13.4 输出能力

- 能将目标 SVG 页面转换为最终 PPTX
- 最终 PPT 中大部分核心图形可编辑
- 静态背景、图片、图标可保留原样

### 13.5 可运维性

- 有清晰日志和错误信息
- 失败任务可追踪到具体阶段和页面
- 中间产物可选保留用于排查

## 14. 风险与对策

### 14.1 风险：模型生成了 `svg_to_pptx` 不支持的 SVG

对策：
- 建立 SVG 子集白名单
- 生成后先校验再转换
- Prompt 严格限制输出语法

### 14.2 风险：模板页结构过于复杂，难以稳定识别编辑区域

对策：
- 优先支持标准模板
- 对复杂模板设置导入校验
- 对导入失败模板返回明确错误

### 14.3 风险：图形可编辑性不如预期

对策：
- 限定架构图和时序图的绘制规范
- 先支持少量稳定图形语法，再逐步扩展
- 输出转换 trace 便于定位失败元素

### 14.4 风险：长任务阻塞 HTTP

对策：
- 统一使用异步任务接口
- 后台执行生成流程
- 通过状态接口轮询结果

产出：

产出：

## 16. 结论

本次重构的本质，不是简单把旧脚本包一层 API，而是把项目升级为：

**以 SVG 为中心的、可异步调用的、可生成可编辑 PPT 的 FastAPI 服务。**

最关键的产品原则有三条：

- **模板主链路从 JSON 转向 SVG**
- **图形主链路从 Mermaid 图片转向可编辑 SVG**
- **接口主链路从脚本执行转向异步任务服务**
